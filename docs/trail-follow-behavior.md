# 軌跡追従挙動メモ

## 修正前の症状

- 実機確認で、ジェスチャー開始瞬間に画面が一瞬止まる感触があった
- 新しいジェスチャー開始時に、直前の軌跡が一瞬再表示されることがあった
- カーソル位置から軌跡が発生する点は前回より改善していたが、開始シーケンスがまだ不自然だった

## 原因

- オーバーレイウィンドウを `Show()` / `Hide()` しながら使っており、最初の表示タイミングで見た目の切り替えコストが出やすかった
- フェードアウト用タイマーと、次回ジェスチャー開始時の非同期描画更新が同じオーバーレイへ流れ込める状態だった
- 前回セッションの pending update を、新しいセッション開始時に明示的に切り捨てる仕組みがなかった
- 開始時 clear と新規描画の順序が、セッション単位では保証されていなかった

## 前回軌跡が見えていた理由

- 前回ジェスチャーの完了後、フェードアウト待ちの状態が残る
- 次のジェスチャー開始時に、前回セッション由来の描画更新または clear timer の影響が新しい開始と競合する
- その結果、前回軌跡が 1 フレームだけ見えてから新しい軌跡へ切り替わる経路があった

## ちらつき / 停止感の原因

- 開始時にオーバーレイウィンドウの表示切り替えが入っていた
- 開始時描画の直前に、前回状態の消去や全画面オーバーレイ準備が同じ UI 経路に乗っていた
- 開始イベントを「新しいセッション」として扱っていなかったため、古い描画更新を抑止しきれていなかった

## 修正内容

- オーバーレイは初期化時に一度だけ表示準備し、その後は `Hide()` せず、描画要素だけを出し入れする方式へ変更
- `GestureTrailUpdateKind` と `SessionId` を導入し、ジェスチャー開始・更新・完了・クリアを明示的に分離
- 新規 `Start` 受信時は以下を必ず実施
  - 前回 clear session の無効化
  - clear timer 停止
  - 既存描画の即時クリア
  - 新セッションを active session として登録
- 古い `SessionId` の更新は、新セッション開始後に描画しないよう無効化
- 新規開始イベントは `DispatcherPriority.Send` で優先的に処理し、開始前の clear を先に通すよう変更

## 修正後の開始シーケンス

1. Trigger ボタン押下
2. しきい値超過まで pending 状態を維持
3. しきい値超過時に `Start` イベント発行
4. overlay 側で前回状態をクリア
5. 新しい `SessionId` を active に設定
6. 今回の点列だけを描画開始
7. 以後は `Update`
8. 解放時に `Complete`
9. フェード完了後に current session のみクリア

## このセッションで確認できた事実

### ユーザー実機で確認された事実

- カーソル位置から軌跡が発生する点は改善している
- ただし開始時の停止感、ちらつき、前回軌跡の再表示が残っていた

### 自動確認で確認できたこと

- `SessionId` ベースの開始・更新・完了・クリア経路を実装した
- `artifacts/run-build/GestureHotkeyApp.exe` 用ビルドを作成した
- U / D 判定は引き続き `UP_RESULT=U`, `DOWN_RESULT=D`

### まだ確認できていないこと

- 修正後の物理マウス実機で、開始時の停止感がなくなったこと
- 修正後の物理マウス実機で、前回軌跡が一瞬も再表示されないこと
- 修正後の物理マウス実機で、開始直後の見た目が自然になったこと

## 今後の調整ポイント

- 実機での再確認
- まだ停止感が残る場合は、overlay bounds 再計算の頻度削減
- 必要なら trail clear 専用の軽量パスを追加
- 必要なら開始 1 フレーム目だけ点数を制限して描画負荷を抑える
